개발 입문 스터디 4주차를 들었다. 마지막 강의였다. 4주차까지 알차게 배웠다고 생각했는데 4시간 수강한거라고 하신 리드님 말씀 듣고 놀랐다. 아무튼 오늘은 협업 방식에서 일어날 수 있는 브랜치 충돌과 commit convention에 대해 자세히 다뤘다. 브랜치 충돌은 협업할 때 충분히 일어날 수 있는 일이라고 생각했는데 convention의 중요성을 새롭게 느낄 수 있었다.
### git flow와 github flow
- git flow
스터디 강의 내용대로 main 브랜치를 hotfix, develop, feature, release 브랜치로 **세세하게 나눈 것이 특징**이다. 버전 관리를 하는데 효과적이고 체계적이다.
- github flow
git flow와는 다르게 feature 브랜치 위주로만 사용한다. 즉, **브랜치를 최소화 하는 것**이다. 그러면서 보다 단순한 구조를 띄고 브랜치 관리도 쉬운 편이다. 그래서 git flow는 대규모 프로젝트를 할 때 사용되고 github flow는 규모가 덜한 프로젝트에서 편리성을 위해 사용된다.
### commit convention
convention이란 **협업에 있어 갖춰야할 예의**와 같다. commit 내용이 자신만 알아볼 수 있다면 그냥 개인 프로젝트를 하는 것이 나을 것이다. 이처럼 규칙 없이 작성된다면 나중에 해당 commit이 어떤걸 수정한 것인지 알아보기 힘들다.
- feat : 새로운 기능 추가
- fix : 버그 수정
- docs : 문서 수정 (코드 수정 X)
- style : 코드 스타일 수정 (코드 동작은 그대로)
- refactor : 코드 구조 개선
- test : 테스트 코드 추가 및 수정
- chore : 기타 작업
- rename : 파일명 변경
- remove : 파일 삭제
형식은 ``` type : 수정 내용 작성 ``` 이다.
**convention은 팀마다, 팀의 프로젝트마다, 레포지토리마다 달라서 세팅을 할 때 미리 정해야 한다.**
